# /flutter-release ‚Äî Cut a SemVer release

## Role
Release manager ‚Äî turn the accumulated changes into a clean, versioned, release-ready changelog.

## Goal
Stamp the pending `[Unreleased]` changelog entries into a dated SemVer version block, bump the version at its single source (and the Android build number), and keep every version location consistent.

## Deliverable
Updated `docs/release/CHANGELOG.md` (a new released block + reset `[Unreleased]`), the bumped version source + mirror + incremented `+N` build number, and a copy-pasteable release note.

---

## Prerequisite

The project must be loaded (`/flutter-load-project`, OR `docs/specs/04-architect.md` / README present). If the project root has not been provided in this flow, first ask: `Project root? (folder path)`.

Load context: `@rules/versioning.md` (+ `@rules/verification.md`).

## Step 1 ‚Äî Read the current state

1. Read `docs/release/CHANGELOG.md`.
   - **Absent** ‚Üí the app predates the changelog system. Stop and route to `/flutter-load-project` (it proposes the retroactive initialization). Do not create it here.
   - Present ‚Üí read the `## [Unreleased]` section and the current top released version.
2. Read the canonical version: `pubspec.yaml` `version: x.y.z+N` ‚Äî capture both the `x.y.z` part and the build number `N`.

## Step 2 ‚Äî Nothing to release?

If `[Unreleased]` has no entry: tell the user there is nothing to release (current version = X.Y.Z) and stop. Do not create an empty version block.

## Step 3 ‚Äî Infer and propose the increment

Apply the inference of `@rules/versioning.md`:
1. Any `**BREAKING:**` marker or a `Removed` entry ‚Üí **MAJOR**.
2. Else any `Added` entry ‚Üí **MINOR**.
3. Else (only `Fixed` / internal `Changed`) ‚Üí **PATCH**.

Compute the three candidate versions from the current one. Ask with `AskUserQuestion` (single question), the inferred increment **first and marked `(recommended)`**:

- `MAJOR ‚Üí <x+1>.0.0` ‚Äî [what counts as breaking, per `@rules/versioning.md`]
- `MINOR ‚Üí <x>.<y+1>.0` ‚Äî new backward-compatible feature(s)
- `PATCH ‚Üí <x>.<y>.<z+1>` ‚Äî fixes / internal changes only

Show the accumulated `[Unreleased]` entries above the question so the user decides with full context. The user has the final call (notably to force a MAJOR the inference cannot detect).

## Step 4 ‚Äî Apply the release

On confirmation, using the chosen version `NEW` (`x.y.z` form) and the session date `<YYYY-MM-DD>`:

1. In `docs/release/CHANGELOG.md`: rename `## [Unreleased]` content into a new block `## [NEW] - <YYYY-MM-DD>` (keep the category order Added/Changed/Fixed/Removed/Security), and insert a fresh empty `## [Unreleased]` above it. The block uses the `x.y.z` form ‚Äî **no `+N`** in the changelog.
2. Bump the **canonical** version: `pubspec.yaml` `version` ‚Üí `NEW+<N+1>`. Keep the `x.y.z` part = `NEW` **and increment the Android build number** `N` ‚Üí `N+1` (Android requires a monotonically increasing build number ‚Äî `@rules/versioning.md`).
3. Sync the **mirror**: `lib/core/config.dart` `appVersion` ‚Üí `NEW` (the `x.y.z` part only, **no `+N`**; must equal the `x.y.z` part of `pubspec.yaml` `version`).
4. Do **not** touch `docs/specs/`, the README, or the app `CLAUDE.md` origin (that records the framework version, not the app version).

## Step 5 ‚Äî Verify + hand off

1. Apply `@rules/verification.md` version consistency: the top released version == the `x.y.z` part of `pubspec.yaml` `version` == `config.dart appVersion`; the `+N` build number was incremented; `[Unreleased]` reset empty.
2. Output the cut version block verbatim in a fenced markdown snippet ‚Äî ready to paste into a GitHub / Play Store release note.
3. Remind the user that committing and tagging are theirs to do (this skill never commits): e.g. `git commit -am "release: vNEW"` then `git tag vNEW`.

## Anti-patterns ‚Äî what NOT to do
- **Do not** create the changelog here if it is absent ‚Äî route to `/flutter-load-project`.
- **Do not** cut a release with an empty `[Unreleased]`.
- **Do not** apply a MAJOR without explicit user confirmation.
- **Do not** bump one version location and forget the mirror (`pubspec.yaml` ‚Üî `config.dart appVersion`).
- **Do not** forget to increment the `+N` build number in `pubspec.yaml`, and **do not** write `+N` into the changelog block or `config.dart appVersion`.
- **Do not** rewrite or reword past released blocks ‚Äî only `[Unreleased]` moves.
- **Do not** commit or tag ‚Äî surface the commands, let the user run them.

## When the user asks something adjacent
- **"Just bump to X.Y.Z"** ‚Üí honor an explicit version, but still move `[Unreleased]` into the dated block, increment `+N`, and sync the mirror.
- **"Release and build the APK/AAB"** ‚Üí do the changelog + version bump; the actual build (`flutter build apk --release` / `flutter build appbundle --release`, `@rules/config.md`) and any commit/tag stay the user's action (surface the commands).
